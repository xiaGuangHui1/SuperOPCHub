## 一、人类中介工作流分析

### 1.1 中介的本质

**中介的核心价值在于：甲方说不清自己要什么，乙方也不主动找甲方，中介桥接这两端的信息断层。** 这不同于信息撮合平台（如淘宝），后者的前提是用户已经知道自己要什么。

### 1.2 通用工作流

```
Step 0: 获取信任（前置）  →  Step 1: 接住痛苦
Step 2: 翻译需求（核心）  →  Step 3: 匹配供给
Step 4: 促成交易         →  Step 5: 兜底善后
```

| 步骤 | 核心动作 |
|------|---------|
| Step 0：获取信任 | 身份锚定、案例展示、口碑背书、零风险承诺 |
| Step 1：接住痛苦 | 倾听不打断 → 锚定情绪 → 建立安全空间 → 降级焦虑 |
| Step 2：翻译需求 | 把破碎、情绪化的表达翻译为结构化需求（场景→范围→约束→偏好，逐层覆盖） |
| Step 3：匹配供给 | 多维匹配：专业能力、预算、档期、风格、地域、口碑、风险，软硬兼顾 |
| Step 4：促成交易 | 转换表达给乙方、对齐双方预期、给2-3选项+明确建议、降低风险 |
| Step 5：兜底善后 | 介入协调、重新对齐、启动退换机制 |

### 1.3 能力模型：人类 vs AI

| 能力维度 | 人类中介 | AI中介 |
|---------|---------|--------|
| 共情能力 | 天然强 | 可模拟，缺真实情感 ✗ |
| 需求翻译 | 依赖经验，不稳定 | 结构化、系统性 ✓ |
| 领域知识 | 深度依赖个人经验 | 海量知识，持续更新 ✓ |
| 供给匹配 | 脑容量有限，偏见严重 | 全量供给库，算法匹配 ✓ |
| 兜底善后 | 灵活，有温度 | 按规则处理，缺灵活度 ✗ |
| 记忆/上下文 | 容易遗忘 | 永不遗忘 ✓ |
| 可扩展性 | 1对少量 | 1对无限 ✓ |
| 一致性 | 情绪疲劳影响 | 始终如一 ✓ |

**结论**：AI在分析性工作（翻译需求、匹配供给）上碾压人类，在关系性工作（共情、兜底）上不如人类。好的AI中介应让AI做它擅长的、把人类优势环节也纳入设计。

## 二、AI中介系统架构设计

### 2.1 系统总览

```
                        用户/甲方 (模糊需求)
                              │
┌─────────────────────────────────────────────────────┐
│                      AI 中介系统                      │
│                                                      │
│  输入层(多模态) → 理解层(意图挖掘/需求结构化/缺失补充) │
│  → 匹配层(供给检索/多维匹配/排序解释) → 对接层(方案推荐)│
│                                                      │
│  信任层(身份/评价/隐私) | 知识层(领域知识/乙方画像)     │
│  善后层(过程追踪/纠纷调解/智能转介)                    │
└─────────────────────────────────────────────────────┘
                              │
                         乙方/供给方
```

### 2.2 核心模块

#### 模块一：输入层 — "什么都能扔进来"

| 输入形态 | 处理方式 |
|---------|---------|
| 自由文本 | LLM直接解析（主通道） |
| 图片/截图 | 视觉模型提取特征 → 结构化 |
| 语音 | ASR → LLM解析 |
| 文件/链接 | 内容提取 → 特征化 |
| 情绪信号 | 情绪识别 → 调整交互策略 |

**多轮采集**（分步来，不一次问完）：
1. 锚定场景："你想解决什么问题？"
2. 理解范围："哪些必须做？哪些可以暂缓？"
3. 量化约束："预算范围？时间要求？"
4. 补充细节："有没有参考案例或偏好？"

#### 模块二：理解层 — "把模糊翻译成精确"

**意图挖掘引擎**：情绪识别 → 关键信息提取 → 缺失项识别 → 隐含假设检测 → 优先级推断。输出结构化需求 + 置信度 + 待确认项。

**领域需求模型**（每垂类构建一次，定义完整需求Schema，用于检测信息缺口）：

```
IT外包：
{
  "项目类型": "小程序/APP/网站/后台系统/SaaS/...",
  "目标平台": ["iOS", "Android", "Web", ...],
  "核心功能": [...],
  "技术栈偏好": "不限/React/Vue/Flutter/...",
  "项目阶段": "从零开发/已有代码改进/维护/...",
  "预算范围": "<5万/5-20万/20-50万/>50万",
  "时间要求": "紧急/正常/不着急",
  "合作模式": "整包/驻场/远程/混合",
  "后续维护": "不需要/需要/待定"
}

家装：
{
  "房屋类型": "新房/二手房/局部翻新/自建房",
  "房屋面积": number,
  "装修范围": "全屋/局部",
  "风格偏好": "现代简约/北欧/日式/工业风/新中式/...",
  "预算范围": "<10万/10-20万/20-50万/>50万",
  "入住时间": date,
  "家庭结构": "独居/情侣/三口之家/老人同住/...",
  "有无宠物": "有/无",
  "特殊需求": "适老化/儿童安全/智能家居/储物/...",
  "服务类型": "纯设计/半包/全包/设计+施工",
  "地理位置": "城市+区域"
}
```

**探询策略**（按用户状态切换）：

| 用户状态 | 策略 | 示例 |
|---------|------|------|
| 信息极度匮乏 | 枚举法 | "可能的情况有A、B、C，哪个最接近？" |
| 信息基本完整 | 验证法 | "我理解大概是X，Y方面再补充一下？" |
| 用户已有偏见 | 对比法 | "方案A可以，但B成本更低能达到同样效果" |
| 情绪激动 | 安抚优先 | "先告诉我发生了什么，我们一起看怎么解决" |
| 需求之间矛盾 | 场景化暴露 | "如果预算只能二选一，你选哪个？" |

#### 模块三：匹配层 — "找到最对的那个人"

**乙方画像 Schema**：
```
{
  基础信息: { 身份: 个人/团队/公司, 规模, 成立年限, 所在地 },
  能力矩阵: { 核心技能: [...], 可服务类型: [...], 擅长行业: [...], 认证/资质: [...] },
  服务状态: { 当前负载: 有空/稍忙/已满, 典型周期, 最小/最大预算, 是否接受远程 },
  质量档案: { 历史项目数, 完成率, 平均评分, 各维度评分, 退款/纠纷率, 代表案例 },
  合作偏好: { 偏好项目类型, 偏好预算范围, 不接受类型 }
}
```

**多维匹配**：匹配分数 = Σ(维度得分 × 权重)，权重由AI根据需求挖掘结果自动推断，不让用户手动调参。不同用户场景权重不同：预算紧张则预算权重高，预算充裕则评价权重高，时间紧急则档期权重高。

**匹配解释**（必须有"为什么推荐"）：每个推荐附带具体理由——项目数匹配、预算匹配、档期匹配、评价关键词。

#### 模块四：对接层 — "让双方说同一种语言"

将用户自然语言翻译为乙方可读的Brief（结构化的任务描述），并主动识别期望偏差（预算vs市价、工期vs典型周期、主观描述vs客观锚点）。

#### 模块五：信任层

| 机制 | 实现 |
|------|------|
| 透明性 | 每一步解释"为什么"，非黑盒推荐 |
| 可溯源 | 评价、案例可查看原始信息 |
| 有退路 | 明确的不满意可换人/退款，AI自动执行 |
| 隐私保护 | 匹配前双方身份脱敏 |
| 渐进信任 | 先小范围试用，满意再深入 |

#### 模块六：善后层

触发条件：用户报不满意、项目逾期N天、评分掉到阈值以下、用户长时间未确认。AI自动收集事实、对比协议、提出调解方案，人工仅在AI无法解决时介入。

### 2.3 完整对话流示例

```
用户："我想做个小程序，大概就是能让用户在上面买东西那种，不用太复杂"
AI："电商类小程序？我先问几个关键问题：
     1. 商品你自己发货还是多商家？
     2. 用户主要用微信还是其他？
     3. 大概多少种商品？"
用户："就我自己卖，二三十种手工皂，主要朋友圈推广"
AI："清楚了，小型手工品自营店铺。"
[显示结构化需求摘要]
"为你找到3位开发者：
🏅 李工工作室 — 做过6个类似项目，报价1.5-3万，2周内上线MVP，评分4.9
⚠️ 预算和'像淘宝'有差距：模板搭建1-2万/2周，全定制3-5万/4周+，你倾向哪个？"
```

## 三、垂类实现一：IT软件外包服务

### 3.1 为什么需要AI中介

| 现状 | AI中介如何改善 |
|------|--------------|
| 甲方说不清技术需求 | "我想要个APP" → 结构化需求文档 |
| 开发者质量参差不齐 | 乙方画像+真实项目评价 |
| 信息不透明 | 公开同类项目预算参考 |
| 乙方也痛苦（需求不清） | 能清楚表达需求的甲方对乙方也是好事 |
| 撮合效率低 | 一次表达，AI统一匹配 |

### 3.2 领域需求模型

```
{
  项目名称, 项目类型: "小程序/APP/网站/桌面软件/后台系统/其他",
  行业: "电商/教育/医疗/金融/社交/企业服务/其他",
  目标平台: "iOS/Android/Web/桌面/跨平台/不限",
  技术栈偏好: string[], 现有技术资产: string,
  开发范围: "从零开发/基于现有修改/功能追加/重写/维护",
  功能复杂度: "简单(1-2核心功能)/中等(3-5模块)/复杂(多模块+多角色)",
  是否需要设计, 是否需要后端,
  预算范围: {min, max}, 预算弹性: "固定/小幅度浮动/按实际谈",
  时间要求: {期望交付, 截止日期},
  合作模式: "整包/按人天/按月/待沟通",
  对乙方期望: string[], 不可接受项: string[], 项目背景
}
```

### 3.3 乙方画像

```
{
  身份信息: { 类型: "个人/工作室/外包公司", 规模, 位置, 可工作模式: ["远程","驻场","混合"] },
  能力矩阵: { 技术栈: [{名称, 熟练度, 项目数}], 服务类型: [{类型, 经验}], 行业经验, 角色覆盖 },
  服务状态: { 当前承接数, 容量: "有空/较忙/已满", 最小/最大项目金额, 可开始时间 },
  质量档案: { 总项目数, 完成率, 评分: {技术质量,沟通,准时,预算控制}, 典型周期, 客户评价标签 }
}
```

### 3.4 领域特有难点

| 难点 | AI应对 |
|------|--------|
| 用户用"产品语言"而非"技术语言" | 拆解类比：把"像XX"拆成具体功能模块让用户选 |
| 用户不知道市价 | 给出类似项目预算参考区间 |
| "简单"没有意义 | 给"简单/中等/复杂"的具体示例锚定 |
| 技术栈选择绑架需求 | 追问选择动机，枚举替代方案 |
| 不区分MVP和完整版 | 主动引导：先做MVP验证，再迭代，预算分期 |

## 四、垂类实现二：家装装修服务

### 4.1 行业结构

```
设计侧：独立设计师、设计工作室、材料供应商、家具厂商
施工侧：装修公司(设计+施工)、工长/包工头、各工种工人(水电木瓦油)

服务模式：
  纯设计：设计师出图，施工自己找人
  半包：装修公司包施工+辅材，主材自己买
  全包：设计+施工+所有材料
```

### 4.2 业主典型痛苦

| 痛苦 | 根因 |
|------|------|
| 不知道该找谁 | 行业碎片化，设计师/公司/工长/独立工人，不知该找哪类 |
| 怕被坑 | 信息不对称极端，材料/工艺/报价全是黑洞 |
| 预算一超再超 | 报价阶段故意漏项，施工中不断增项 |
| 效果图和实际差太远 | 设计/施工/材料三方脱节，无人兜底 |
| 工期一拖再拖 | 不同工种衔接靠工长协调，一个环节延误全盘后延 |

### 4.3 领域需求模型

```
{
  房屋基础: { 类型: "新房毛坯/二手房翻新/精装微改/局部改造/自建房", 面积, 户型, 位置: {城市, 区域}, 交房状态 },
  装修范围: { 范围类型: "全屋/局部", 涉及空间: [...], 是否拆改, 是否换门窗 },
  风格偏好: { 风格: "现代简约/北欧/日式/新中式/美式/工业风/轻法式/混搭/没概念", 色调: "暖色/冷色/中性/无所谓", 参考图 },
  家庭成员: { 家庭结构: "独居/夫妻/三口之家/多代同堂", 有老人, 有小孩, 有宠物 },
  特殊需求: { 智能家居, 储物需求, 适老化, 儿童友好, 在家办公, DIY意愿 },
  预算时间: { 预算: {min,max}, 预算类型: "硬装+软装全含/仅硬装/仅软装", 入住时间, 时间弹性 },
  服务偏好: { 服务模式: "不确定/纯设计/半包/全包", 价值偏好: "性价比/品质/设计/省心优先" }
}
```

### 4.4 乙方画像（多角色组合）

```
{
  类型: "独立设计师/设计工作室/装修公司(整装)/工长施工队/定制家具/监理",
  基础信息: { 从业年限, 团队规模, 服务区域: [...] },
  风格专项: [{ 风格, 项目数, 代表案例 }],
  项目能力: { 可承接类型, 最小/最大预算, 月承接量, 典型工期 },
  质量档案: { 总项目数, 评分: {设计,施工质量,预算控制,工期,沟通}, 投诉率, 效果图还原度 },
  价格参考: { 设计费单价(元/平米), 半包报价, 全包报价 }
}
```

### 4.5 领域特有难点

| 难点 | AI应对 |
|------|--------|
| "温馨"无法量化 | 用风格配图让用户视觉选择 |
| 用户不知道全套流程 | 生成可视化全流程路线图 |
| 预算黑洞 | 展示同类房屋/风格的实际花费参考 |
| 需求随了解而改变 | 允许迭代，不要求一次定死 |
| 服务模式不明 | 先花时间解释三种模式区别，再让用户选 |
| 地域强相关 | 匹配必须优先过滤地域 |

### 4.6 对话设计：从模糊到方案

```
阶段1：理解房屋 — "什么类型？多大面积？几室几厅？"
阶段2：探索生活方式 — "几人住？有老人小孩？常做饭？有宠物？" → 推断隐含需求
阶段3：风格发现 — 展示6组实景图，凭感觉选，不用纠结术语
阶段4：预算对齐 — 展示同户型不同档次花费参考
阶段5：明确服务模式 — 解释纯设计/半包/全包的区别，基于用户情况给建议
阶段6：生成需求摘要 + 匹配推荐
```

### 4.7 编程视角

```python
# 装修项目 = 一组顺序依赖的任务
装修项目 = [
    设计阶段: [量房, 平面方案, 效果图, 施工图],
    拆改阶段: [拆墙, 砌墙, 铲墙皮],
    水电阶段: [定位, 开槽, 布管, 穿线, 验收],
    泥瓦阶段: [防水, 贴砖, 找平],
    木工阶段: [吊顶, 柜体, 造型],
    油漆阶段: [批灰, 打磨, 底漆, 面漆],
    安装阶段: [橱柜, 门, 地板, 灯具, 洁具, 开关面板],
    软装阶段: [家具, 窗帘, 装饰]
]
# 关键约束：每个阶段依赖上一个，无法并行
# 供给方 = 多类型角色组合: Designer + GeneralContractor + 各工种 + 材料供应商 + Inspector
```

## 五、跨领域对比与通用模式

### 5.1 两个垂类对比

| 维度 | IT外包 | 家装 |
|------|--------|------|
| 需求模糊度 | 中等 | 高 |
| 供给复杂度 | 单一角色 | 多角色组合 |
| 是否可线上闭环 | ✓ | ✗ 必须线下 |
| 地域依赖 | 弱 | 强 |
| 单位价格 | 万-十万级 | 十万-百万级 |
| 频率 | 低频 | 极低频(一生1-2次) |
| AI最大价值点 | 需求翻译(产品语言→技术语言) | 全流程引导+多角色协调 |

### 5.2 通用 vs 领域特有

```
通用层（跨领域可复用）：
  意图挖掘引擎、多轮探询策略、匹配+排序+解释、期望对齐
  信任与评价体系、善后/兜底机制、对话流状态管理、维度权重推断

领域层（每垂类定制）：
  需求模型Schema、乙方画像模板、匹配维度定义、典型对话脚本
  预算参考区间、风格/类型枚举、行业名词库、规则/忌讳列表
```

### 5.3 AI vs 人类的差异化优势

1. **永不疲惫**：深夜焦虑用户能得到即时耐心回应
2. **全量供给库**：覆盖全市场供给，实时更新乙方状态
3. **无利益相关**：算法匹配非回扣驱动，可公开匹配逻辑
4. **全流程记忆**：记得用户三个月前提到的一切
5. **知识无盲区**：可接入多个垂直领域知识库
6. **规模化**：同时服务成千上万用户
7. **透明计算**：拉取同类项目实际成交数据做对比

### 5.4 AI仍需人类的地方

1. 用户需要"被理解"的感觉 → AI做前期挖掘，关键时刻转真人顾问
2. 复杂纠纷调解 → AI收集事实、生成草案，人类做最终裁定
3. 审核类工作 → AI初筛，人工深度审核
4. 非标灵活应对 → AI识别 "I don't know" 并转人工
5. 信任感 → AI+人类混合模式，明确告知背后有人兜底

### 5.5 核心原则：必须给出建议

**区分"好中介"和"搜索引擎"的标准：AI必须给出明确建议，用户只需要"同意"或"不同意"。**

| 场景 | 不该问 | 应该给 |
|------|--------|--------|
| 平台选择 | "你选小程序还是APP？" | "建议做小程序，获客成本低100倍。不同意可选APP。" |
| 预算 | "预算是多少？" | "同类市价3-8万，基于你的功能建议5万。嫌贵我告诉你怎么砍功能。" |
| 技术选型 | "用React还是Vue？" | "建议Flutter跨平台，一套代码兼顾两端。有Vue团队另说。" |
| 服务模式 | "半包还是全包？" | "以你'没太多时间'来看，建议全包。多花3-5万省你几十个周末。" |

**AI比人类更敢给建议**：人类中介怕追责、怕得罪返佣关系、怕经验不足下错判断。AI的建议有数据支撑、可追溯、无利益偏见，应成为"世界上最敢于给明确建议的中介"。

**建议格式**：我建议[明确选择] → 理由1/2/3 → 如果不同意，备选A的代价、备选B的代价。

**唯一例外**：审美/价值观/个人偏好属于用户自我，AI只提供结构化选项帮助发现偏好，不替用户决定。

### 5.6 核心原则：少问问题，多得信息

矛盾：问题多了用户流失，信息不够匹配不准。解法：**从用户已说的信息中推断大量没说出的信息。**

#### 六大策略

**策略一：隐式推理**（从"怎么说"推断"要什么"）

| 信号 | 推断 |
|------|------|
| 反复强调"简单""别复杂" | 预算紧张、怕过度设计 |
| "烦死了""搞了一个月" | 挫败状态，先安抚 |
| 用准确术语 | 有领域知识，可高效沟通 |
| 描述大量细节 | 已想清楚，需要执行者非参谋 |
| 主动提截止日期 | 时间约束强，档期权重提高 |
| "预算不能超过" | 预算约束硬，匹配保守报价 |

**策略二：默认假设**（用户省略的信息用默认值覆盖，错了用户纠正）

来源A：同类用户画像 → 默认（如"个人创业者首次做产品" → 默认：预算敏感、要快、首选小程序）

来源B：领域统计先验 → 默认（如IT外包默认用云服务、家装默认800×800地砖）

规则：假设标注在确认摘要里，用户只需纠正不对的地方。如果10维中7个用默认覆盖，用户只纠正1-2个，等于3个问题得到10维信息。

**策略三：信息收益排序**（先问最能缩小范围的）

```
项目类型 → 排除~70%乙方 ★★★★★
预算     → 排除~60%乙方 ★★★★★
平台     → 排除~40%乙方 ★★★★
行业     → 排除~30%乙方 ★★★
需设计   → 排除~20%乙方 ★★
```

**策略四：二选一压缩**（把构造性问题变选择性问题）

认知心理学：识别(recognition)永远比回忆(recall)容易。不给选项=让用户做填空题，给选项=做选择题。如"你的预算是？" → "这类项目三个价位：3-5万/8-15万/20万+，你在哪一档？"

**策略五：行为信号**（用点击投票，不开口就拿偏好）

| 行为 | 推断 |
|------|------|
| 反复看某乙方案例 | 风格偏好信号 |
| 点开"预算参考"页 | 价格敏感 |
| 快速划过前3推荐 | 前3都不对，需调整匹配 |
| 收藏某案例 | 强兴趣，增加同类权重 |

**策略六：先匹配再修正**（粗糙匹配后靠反馈收敛）

不等信息齐全：问2-3高收益问题 → 粗糙匹配10个结果 → 用户反馈"都不太对" → 缩小范围 → 精准匹配。每次反馈等于用户回答了额外问题而不自知。

**应用效果**：10个维度仅需3个问题，剩下7个靠推断和默认假设填满。

### 5.7 核心原则：本地信号采集 — 不开口就拿数据

用户不开口不代表没信息。人类中介见面能看到穿着、表情、时间，AI线上看不到这些——但AI有传感器、HTTP头、浏览器API能补回。

#### 信号四象限

```
                        无需权限              需要授权
          ┌─────────────────┬──────────────────┐
被动      │ IP→城市          │ GPS→精确位置     │
          │ User-Agent      │ 蓝牙/WiFi扫描    │
          │ 时区/语言        │                  │
          │ 屏幕分辨率       │                  │
          │ Referrer        │                  │
          ├─────────────────┼──────────────────┤
主动      │ 停留时长         │ 文件上传          │
          │ 滚动深度         │ 通讯录            │
          │ 打字速度         │ 日历/相册         │
          │ 点击热力图       │                  │
          └─────────────────┴──────────────────┘
```

**优先级**：左上角（被动×无需权限）100%采集，零摩擦零隐私风险。右下角只在用户拒绝回答关键问题后才请求。

#### 零摩擦信号清单

| 信号 | 获取方式 | 推断 | 影响 |
|------|---------|------|------|
| IP→城市 | HTTP头 | 所在城市 | 家装直接过滤本地乙方 |
| 访问时间 | Date.now() | 凌晨=焦虑型 | 先安抚再挖掘 |
| User-Agent | navigator | 设备/OS | 优先匹配同平台开发者 |
| 屏幕分辨率 | window.screen | 设备档次 | 大屏→办公需求 |
| 语言偏好 | navigator.language | 母语 | 匹配同语言乙方 |
| Referrer | document.referrer | 来源 | 知乎=做过功课；搜"急"=焦虑 |
| Cookie/LocalStorage | 本地读取 | 是否老用户 | 跳过已问问题 |

#### 典型应用场景

1. **免去"你在哪"**：读IP→推断城市→直接匹配本地乙方，0轮对话
2. **设备泄露偏好**：用户用iPhone 15说"做APP"→默认先iOS端
3. **时间推断紧迫度**：凌晨02:02发消息→焦虑型，不深挖，早上推送结果

#### 隐私硬线

```
✅ 可采：IP→城市(到区县)、设备型号/OS(不含唯一ID)、时间/语言/屏幕、站内行为(不跨站)
❌ 不采：浏览器指纹、剪贴板、已安装应用列表、社交画像(未授权)、GPS精确位置(未授权)
```

**原则**：如果我告诉用户我用了这个信号，他会觉得被冒犯还是被帮助？

#### 信号与问题的关系：替代而非叠加

理想比例：纯对话20+问题 → 混合模式：5-8个问题 + 8-10个本地信号 + 5-8个默认假设 = 同等信息量。

### 5.8 合规性分析

#### 个人信息判断

```
非个人信息：屏幕分辨率、浏览器版本、时区偏移
个人信息(合规可取)：IP地址、User-Agent(含版本号)、浏览历史、点击行为
不可取(未明确同意)：浏览器指纹、Cookie ID、LocalStorage持久ID
```

#### 每个信号合规分析

| 信号 | 是否个人信息 | 可否自动采集 | 注意事项 |
|------|------------|------------|---------|
| IP地址 | 是 | ⚠️ 需告知 | 只取城市级，不存完整IP |
| User-Agent | 灰色 | ✓ 可采 | 单独用非个人信息，组合后可能可关联 |
| 时区/语言 | 否 | ✓ | 无风险 |
| 屏幕分辨率 | 否 | ✓ | 无风险 |
| Referrer | 灰色 | ⚠️ 建议匿名化 | 只取来源域名，不取完整搜索词 |
| 访问时间 | 否(不关联用户) | ✓ | 只做实时策略，不存储 |
| Cookie(会话恢复) | 是 | ✗ 需弹窗 | 功能型可用"合同必需"例外 |
| 行为数据 | 是 | ✗ 需告知 | 隐私政策披露"收集交互数据改善匹配" |
| Referrer搜索词 | 是(可能含敏感) | ✗ 不建议 | 可能获取高敏个人信息 |

#### 三种合法路径

1. **合同必需**(不需弹窗)：如"为匹配本地乙方需知道你所在城市"，只取IP城市级，解析后丢弃完整IP
2. **合法利益**(不需弹窗，需隐私政策披露)：如"检测设备类型以优化匹配精度"
3. **明确同意**(必须弹窗)：如"记录浏览偏好以恢复进度"

#### 建议合规架构

```
采集层              处理层              存储层
IP地址          →  解析出城市        →  丢弃IP，只留城市(内存，不落盘)
User-Agent      →  提取OS+设备类型   →  丢弃完整UA(浏览器端提取)
时间戳           →  判断时段          →  不存储
屏幕分辨率       →  判断设备档次      →  不存储
Referrer        →  提取来源域名      →  不存储(匿名化后丢弃完整URL)
```

**核心策略**：能内存的不落盘，能匿名的不关联身份，能本地的不传服务端。

#### 合规决策树

```
采集信号 →
  ├─ 不做也能匹配？→ 能 → 不采
  └─ 不能 → 能识别到具体人？
       ├─ 不能 → 最小必要采集，隐私政策披露
       └─ 能 → 能否降低精度？
            ├─ 能 → 匿名化后使用
            └─ 不能 → 法理依据？
                 ├─ 合同必需 → 告知后采集
                 ├─ 合法利益 → 披露后可反对
                 └─ 只能靠同意 → 弹窗，可拒绝
```

### 5.9 架构决策：传统API还是大模型

**分工原则**：结构化→结构化用传统API；非结构化自然语言理解/生成用大模型。

```
✓ 传统API/库：
  city = geoip.lookup(ip)              // IP→城市，数据表查询
  os = ua_parser.parse(user_agent)     // UA→OS，规则匹配
  device_type = classify_screen(w,h)   // 分辨率→类别，if/else

✓ 大模型：
  intent = llm.infer("用户说想做个像淘宝但简单点的")
  response = llm.generate("解释为什么建议小程序而非APP")
```

| 任务 | 用谁 | 原因 |
|------|------|------|
| IP→城市 | 传统API | 纯数据表查询，0推理 |
| UA→OS | 传统库 | 规则匹配，毫秒级 |
| 分辨率→设备档位 | 前端if/else | 不需要后端 |
| Referrer→来源 | URL解析 | 字符串截取 |
| 时间→时段 | 前端Date+规则 | if/else |
| 用户描述→需求Schema | **大模型** | 需要语义理解 |
| 需求Schema→匹配结果 | 传统后端 | 索引查询+多维加权 |
| 匹配结果→用户解释 | **大模型** | 需要自然语言生成 |
| 反馈→匹配调整 | **大模型** | 需要语义理解 |
| 对话状态管理 | 传统后端(状态机) | 明确的状态转换规则 |
| 情绪/紧迫度判断 | 混合(规则预判+LLM验证) | 简单规则初筛，LLM细分类 |

**Token成本**：一次IP查询用LLM浪费2000倍时间；每天10万请求用全部LLM做信号解析=6万/年。但需求翻译用传统方案准确率天花板低，该花的token一毛不省。

**传给大模型的输入**是预处理后的结构化上下文，不是裸数据：
```json
{
  "user_message": "...",
  "environment": {"city": "杭州", "device_type": "desktop", "time_period": "work_hours", ...},
  "history": {"last_session": "...", "previous_searches": [...]}
}
```

### 5.10 市场验证

2025-2026年已出现密集产品发布，验证了融合架构方向：

| 产品 | 做法 | 重合度 |
|------|------|--------|
| Metty.ai (2025.5) | 自然语言描述→AI匹配验证过的服务商 | 高度重合 |
| Clutch ChatGPT App (2026.4) | B2B服务商数据接入ChatGPT | 场景重合 |
| HumanX MatchX | 多Agent系统做语义匹配+推理 | 架构重合 |
| GLG (2026.5) | Agentic AI做"问题→结构化研究→专家推荐" | 流程重合 |

**共同特征**：大模型在对话层，传统数据库在匹配层，通过结构化接口连接——与5.9节架构一致。

**我们还多做但市面还缺的**：多轮深度需求挖掘(像人类中介追问"为什么")、跨垂直领域通用框架、AI主动给建议而非把问题抛回、本地信号采集+对话挖掘深度整合、善后兜底机制。

**结论**：融合架构大方向已被验证，但"像人类中介一样深度挖掘"的产品体验还没人做到位。市面是1.0，我们的设计是2.0。

### 5.11 一句话总结

> **AI中介要做的不是取代人类中介的"人情味"，而是用系统能力把"分析、匹配、记忆、透明"做到人类无法企及的水平。模糊表达进来，精准匹配出去。**

## 六、对话体验设计

前五章解决"能不能做"，本章解决"用户愿不愿意聊"。

### 6.1 开场前30秒

标准开场"你好，我能帮你做什么"是自杀式设计——用户来找你就是不知道该怎么描述。

**三种有效开场**：

**A. 展示理解力**：利用本地信号。"你在杭州，用Mac。如果你是来找外包开发团队的，我可以帮你。或有其他需求直接说。"

**B. 降低开口门槛**：给起点按钮——[找人做小程序] [装修房子] [做品牌设计] [其他需求]，识别比回忆容易。

**C. 用例子引导**："比如有人跟我说'想做个类似小红书的产品'，我会帮他拆成图文信息流、用户系统、搜索推荐等模块。你现在有什么想做的？随便说。"

**核心原则**：用户第一条消息的成本应最低——一个词、一个按钮、一句话。

### 6.2 对话节奏

```
开场(前2轮)：慢，信息少 → 用户还没信任，信息轰炸=劝退
探询(中段)：快，一问一答 → 用户进入状态，高效收集
建议(末段)：慢，信息多 → 最终决策需要完整信息消化

快信号(可推进)：回答简洁具体、完整句子、主动补充细节
慢信号(该暂停)："嗯""对"、回答越来越短、长时间不回复
AI应做的事：检测慢信号→主动总结进展问"要继续还是先看结果"
```

### 6.3 信息呈现

**原则1：结构化呈现，不堆砌段落**。大段文字用户跳过，卡片化呈现用户会看。

**原则2：用视觉辅助做选择题**。装修风格直接展示实景图，一眼就知喜不喜欢，比文字描述认知负担趋零。

**原则3：渐进披露，不一次灌满**。
- 第一层：名称 + 关键标签 + "查看详情"
- 第二层：代表案例 + 评价 + 报价明细
- 第三层：完整案例集 + 合同模板

每层只给当前决策需要的信息量。

### 6.4 输入体验

| 通道 | 场景 |
|------|------|
| 自由文字 | "我想……"自然表达 |
| 快速回复按钮 | 封闭式问题，点一下比打字快10倍 |
| 语音输入 | 边走边说、开车时 |
| 图片输入 | 一张参考图比描述省1000字 |
| 文件上传 | 合同/预算表/需求文档 |

**快速回复按钮**：每次只给3-4个选项，通过分层降低认知负荷。第一层[简洁干净][温暖质感][不确定] → 选了再展开第二层。

### 6.5 容错设计

**核心原则**：用户纠正的成本应和AI犯错的严重程度成正比。

| 错误类型 | 修复策略 |
|---------|---------|
| AI理解偏了(用户说小程序，AI回APP) | AI先认错，局部修正，不重来 |
| 用户中途改主意 | 增量更新，保留之前所有有效信息，只改变了的字段 |
| 用户说"都不太对" | 先定位问题维度[价格/风格/不专业/说不清]，再调整匹配 |

### 6.6 进度可见性

在对话自然转折点插入进度卡：
```
✅ 基础需求已清晰
✅ 平台和功能范围已确认
⏳ 正在了解预算和周期
⬜ 匹配最合适的乙方
⬜ 生成推荐方案
大约还需2-3分钟
```

用户知道已完成什么、还要做什么、大概还要多久——不会在每一轮都担心"还要聊多久"。

### 6.7 语气和人格

角色定位：**值得信赖的顾问**（非客服、非销售、非百科）

- ✗ "您好，很高兴为您服务" — 客服腔
- ✗ "根据大数据分析，最优解是" — AI腔
- ✓ "我建议你选小程序，以你的预算和场景，APP获客成本太高" — 有判断、有理由、有人味

关键习惯：
- 不和稀泥("各有利弊请自行选择") → 有判断("建议选A，以你的情况省3周")
- 不假装全能 → 诚实有限("北京做北欧风小户型的靠谱设计师不多，找到3个不错")
- 不甩锅("这取决于你的预算") → 给参照("杭州这边类似项目大概3-5万")

### 6.8 沉默时的主动

| 原因 | 应对 |
|------|------|
| 在思考(光标闪烁未发送) | 超过30秒："不着急慢慢想，也可以随便说" |
| 不知怎么回答 | 2分钟后："刚才问题可能不好答？我换个问法" |
| 不感兴趣了 | 总结性消息："关键点已记下，随时回来继续" |
| 被打断(切到其他应用) | 回来时："上次聊到……要继续还是直接推荐？" |

### 6.9 结束体验

离开时最后一刻感受决定会不会回来。

三个交付物：
1. 可复制/分享的需求摘要(用户拿去直接用的方案起点)
2. 明确的下一步行动(点一下就能联系乙方)
3. 回来的理由("你的偏好我记住了")

### 6.10 体验检查清单

```
□ 用户第一句话就知道产品干什么吗？(3秒测试)
□ AI回复是推进还是挡路？
□ 有进度标记吗？
□ 推荐结果是卡片不是大段文字？
□ 有快速回复按钮？3次点击内完成一轮选择？
□ AI误解后，用户改一个字还是重说一遍？
□ 用户沉默30秒后会发生什么？
□ 退出隔天回来是继续还是重来？
□ 离开前最后看到的东西有没有"搞定了"的感觉？
□ 有没有东西可复制带走(不是截图)？
```

## 七、需求自觉察：帮用户看明白自己

前面解决"AI怎么理解用户"，更深层挑战：**很多时候用户自己也不理解自己**。用户说"我要X"不是因为X最好，而是因为X是他能想到的唯一选择。

### 7.1 三个认知障碍

1. **锚定效应**："我想要APP"→因为朋友做了APP，APP是他知道的唯一形态。实际需要的是"用户能方便找到我的服务"。
2. **抽象迁移困难**："我想要北欧风"→在小红书看过觉得好看，但不知自己养猫、常做饭和北欧风有冲突。
3. **后果盲区**："我全都要"→不知后果是预算翻3倍、工期翻2倍、维护成本翻10倍。

### 7.2 六种技术工具箱

#### 技术一：同路人参照（"跟你一样的人最后选了……"）

解决锚定效应。展示同类用户的三种选择路径，用户从"不知有什么选择"变"在别人走过的路里选"，认知负担量级降低。

```
"之前跟你类似的人，70%选了模板化(1-2周/5千-1万)，20%选了定制(3-4周/3-5万)，10%选先社群再工具。你更接近哪种？"
```

#### 技术二：代价前置（"你想要的功能加在一起是这样的"）

解决后果盲区。实时展示功能累积的代价，不需要AI说"不行"，数字会说话。

```
"你选了9个功能 → 估价12-18万(均值15万) → 开发10-15周 → 月维护3000-5000元
 ⚠️ 你最初提的预算是3-5万。要回到5万内需砍5-6个功能。我帮你排序哪些最有价值？"
```

#### 技术三：强制牺牲（"如果必须砍掉一个……"）

解决优先级模糊。通过多轮二选一暴露真实优先级，不给用户排序任务(认知负担高)，只做二选一(直觉瞬间)。

```
"如果只能保一个：[功能要多5个功能做精] vs [体验要好3个功能极致]"
用户选"体验" → "再砍：[快上线] vs [预算控制住]"
```

#### 技术四：后果预演（"如果你选了A，三个月后是什么样的"）

解决抽象迁移困难。用叙事时间线把抽象选择变成具体场景，用户立刻知道"这是我想要的"或"不是"。

```
"全包模式下第1周：设计师量房你只需1小时见面。第2-3周：手机上确认设计图。
 第4周：施工队进场，项目经理会协调。第12周：拿钥匙。全程你只需签合同+确认设计+3-5次看工地。"
```

#### 技术五：反事实追问（"如果你不做这个，最差会怎样"）

解决恐惧驱动决策。把模糊恐惧具体化，用户往往发现最差情况并非灾难，于是敢做更聚焦的选择。

```
"如果第一版只做iOS，最坏情况是什么？→ Android用户用不了 → 你的目标用户里Android占多少？
 → 第一版目的是验证需求 → iOS数据够不够验证？→ 如果够，先不上Android。"
```

#### 技术六：偏好暴露（"你更讨厌哪种情况"）

利用损失厌恶(强度是获得喜悦的2倍)。问"你更怕什么"比问"你喜欢什么"更真实。

```
"更讨厌：A.装完太素没个性像出租房 vs B.装完太花半年就腻了"
选A → 宁愿冒险过头也不寡淡 → 不是北欧风用户，可能是轻奢或混搭
```

### 7.3 使用时机

```
需求模糊期(前几轮)  → 同路人参照、偏好暴露    (需要参照锚定方向)
需求展开期(中段)    → 代价前置、后果预演       (需要看见累积后果)
需求矛盾期(末段)    → 强制牺牲、反事实追问      (需要暴露优先级)
```

### 7.4 共同原理

用户说不清要什么不是表达能力差，而是缺三种东西：参照、约束、视角。六种技术分别给这三样。

## 八、思维引导：帮用户学会怎么想这个问题

需求自觉察假设用户脑子里有答案只是埋得深。但很多时候用户脑子里根本没答案，因为**不知道这个问题应该怎么想**。AI中介可以把一个领域的思考框架直接借给用户。

### 8.1 需求自觉察 vs 思维引导

- **自觉察**：答案层面帮助（"你要的是X不是Y"），用户在黑暗中摸索，帮他摸到答案
- **思维引导**：过程层面帮助（"这个问题应该从这几个角度想"），用户不知往哪个方向摸索，先给一张地图

### 8.2 为什么需要引导

1. **领域没有地图**：用户不知考虑顺序（风格→预算→功能→材料→施工），上来就问"多少钱"。AI有完整地图，可以先说"花2分钟搞清几件事，价格答案自然有了"
2. **被情绪劫持**：用户说"我就要尽快上线"可能是老板催得焦虑，真正约束非时间。AI先接住情绪再列真实约束
3. **在错误的分辨率上思考**：用户说"想做电商平台"(10000英尺)，需要有人带他到100英尺拆解(支付选什么？注册方式？SKU几种？)

### 8.3 五种思维引导技术

#### 技术一：框架前置 — "先定怎么聊"

不给框架直接问问题，用户会恐慌"我连基本问题都答不上来"。先给框架："预算不是'一个数字'，而是三个：硬边界(绝对不能超)、舒适区(合理不心疼)、扩展区(多花有明显回报你愿意考虑)。有三个数字的人其实硬边界是8万。"

#### 技术二：注意力引导 — "先不看这个"

用户在小程序vs APP纠结10分钟陷入分析瘫痪。不要继续提供利弊信息，引导注意力："先不选平台。核心功能都是内容社区对吧？先把功能聊清楚。如果在5个以内小程序就够，超过10个才需要APP。让功能告诉你答案。"

#### 技术三：降维 — "这个问题拆成三个来判断"

用户说"不知道项目靠不靠谱"——一个无法直接回答的问题。帮他拆：A.需求靠谱吗？(群里有接龙组织活动吗？) B.你能做吗？(你自己就是用户知道痛点) C.能赚钱吗？(现在判断不了)。A和B有答案→结论：值得做，先做MVP。

#### 技术四：托底 — "最坏的情况是什么，我能接受吗"

把模糊恐惧具体化。自己做最坏=花3个月做了一个不能用的东西，可逆。外包最坏=花5万拿到不能用的代码，部分可逆。两种最坏哪个更接受？时间不值钱→自己做，时间值钱→外包。

#### 技术五：进度叙事 — "你现在在这里，接下来到这里，最终到那里"

用户的不确定性焦虑来自"不知道接下来会发生什么"。在对话早期给完整路线图(现在→需求梳理→方案选择→初步沟通→开始合作)，用户大脑从"我需要解决巨大未知"切换到"我只用完成第一步"。

### 8.4 五种技术的配合

```
完全不知道从哪开始  → 进度叙事(先给地图)
卡在具体问题上      → 注意力引导 + 框架前置(移开卡住的问题)
面对复杂决策不知所措 → 降维 + 托底(拆小+具体化恐惧)
纠结做还是不做      → 托底 + 框架前置(列最坏情况+帮用户看自己类型)
```

### 8.5 引导的边界

不该引导时：用户已有清晰方向只在确认细节、在做审美判断、在做价值判断、明确说"我就想自己想想"。引导的前提是用户陷入认知困境，目的不是替他做决定，是给他一套自己能用的思考工具。

### 8.6 引导用户提问

全程被引导可能产生被动接受感。更根本的是：用户最终要带自己的判断做决定，而判断力来源于用户自己生成的问题。

**四个效果**：
1. 从"被诊断"变"共同诊断"：插入"聊到这你脑子里的第一个疑问是什么"→前额叶被激活进入思考状态
2. 用户提的问题就是他的决策权重：用户主动问"一般多久能看到效果"→真实最高权重=速度
3. 提问过程就是自我梳理：能问出问题说明已消化信息找到认知缺口
4. 对自己产生的想法忠诚度更高："我通过问问题搞清楚了"比"AI帮我搞清楚了"留存强

**三级引导**：
- 低阶：帮翻译碎片→问题("你其实在想预算靠不靠谱对吗？")
- 中阶：给选项("你最想先弄明白：[预算靠谱吗] [要准备什么] [最大坑在哪]")
- 高阶：角色互换("如果你是我，你觉得哪里最可能有疑问？")

**插入时机**：刚做完需求摘要、刚给出推荐、用户犹豫不决、对话将结束。不在结束时问"还有什么问题吗"(客服腔)，在张力最大节点问。

## 九、大模型实现方案

### 9.1 各环节分工

```
1. 本地信号采集       → 传统后端 (0 token)
2. 意图挖掘           → 大模型 (语义理解)
3. 需求结构化         → 大模型 (信息抽取)
4. 对话策略决策       → 大模型 (上下文推理)
5. 需求→匹配参数      → 传统后端 (规则转换)
6. 乙方匹配排序       → 传统后端 (向量检索+加权)
7. 匹配解释生成       → 大模型 (自然语言生成)
8. 反馈→匹配调整      → 大模型 (语义理解)
9. 对话状态管理       → 传统后端 (状态机)
```

**大模型本质**：非结构化自然语言 ↔ 结构化数据的转换器。

### 9.2 System Prompt

**约束一：角色定位**
```
你是AI中介，帮助用户向专业服务商表达需求并匹配乙方。
角色不是客服、不是销售，是值得信赖的顾问：
- 帮用户理清真正需要什么（非他最初说的）
- 基于领域知识给出有判断力的建议（非各打五十大板）
- 透明解释每个推荐理由
- 不知道时诚实说不知道
```

**约束二：行为约束**
```
1. 需要的信息主动去拿，不等用户给完整Brief
2. 一次只问1-2个问题，不列清单
3. 用户说不清是正常的，帮它变清楚
4. 用户情绪化先共情再分析
5. 做判断处给建议并解释理由，不把决策负担甩回用户
6. 不确定先确认再推进
```

**约束三：结构化输出**
```
获得足够需求信息时调用update_requirement，不等所有信息齐了才调。
信息足够匹配时调用finalize_requirement。
```

### 9.3 Function Calling 驱动对话

对话流不靠Prompt写死，而是靠不断更新的需求Schema驱动：
1. 用户消息 → 大模型(System Prompt + 历史 + 当前Schema状态)
2. 大模型返回：纯文本(继续对话) 或 update_requirement(新信息) 或 finalize_requirement(该匹配了)
3. 如果update → 更新Schema → 后端检查缺哪些字段 → 注入下一轮上下文
4. 循环

**Function定义**：

```json
{
  "functions": [
    {
      "name": "update_requirement",
      "description": "用户提供新需求信息时调用，不等信息齐全",
      "parameters": {
        "project_type": {"enum": ["小程序","APP","网站","桌面软件","后台系统","其他"]},
        "industry": {"enum": ["电商","教育","医疗","金融","社交","企业服务","其他"]},
        "platform": {"enum": ["iOS","Android","Web","跨平台","微信小程序","不确定"]},
        "budget_range": {"properties": {"min": "number", "max": "number"}},
        "timeline_weeks": "number",
        "complexity": {"enum": ["简单","中等","复杂"]},
        "key_features": {"items": "string"},
        "needs_design": "boolean",
        "needs_backend": "boolean",
        "project_nature": {"enum": ["个人创业","公司项目","side project"]},
        "urgency_level": {"enum": ["紧急","正常","不着急"]}
      }
    },
    {
      "name": "finalize_requirement",
      "description": "所有必要信息已收集完毕，可进行乙方匹配",
      "parameters": {"summary": "string"}
    },
    {
      "name": "clarify",
      "description": "用户表达模糊需先确认再推进",
      "parameters": {"questions": {"items": "string", "description": "需要澄清的问题(1-2个)"}}
    }
  ]
```

### 9.4 对话循环代码

```python
class AIIntermediary:
    def __init__(self):
        self.requirement = {}          # 不断更新的需求Schema
        self.required_fields = [...]   # 垂类必须收集的字段
        self.state = "collecting"      # collecting | matching | recommending

    async def process_message(self, user_message: str, env_signals: dict):
        context = self._build_context(user_message, env_signals)
        response = await llm.chat(
            system_prompt=SYSTEM_PROMPT,
            messages=context["history"],
            functions=ALL_FUNCTIONS,
            function_call="auto"
        )
        if response.has_function_call():
            return await self._handle_function_call(response)
        else:
            return response.text

    def _build_context(self, user_message, env_signals):
        return {
            "history": self.conversation_history[-20:],
            "requirement_state": {
                "collected": self.requirement,
                "missing": self._get_missing_fields(),
                "next_to_ask": self._suggest_next_question()
            },
            "environment": env_signals
        }

    def _get_missing_fields(self):
        high_priority = ["project_type", "budget_range", "project_nature"]
        return [f for f in high_priority if f not in self.requirement]

    async def _handle_function_call(self, response):
        func_name = response.function_name
        args = response.function_args
        if func_name == "update_requirement":
            self.requirement.update(args)
            if self._is_ready_to_match():
                return await self._trigger_matching()
            else:
                return await self._continue_conversation()
        elif func_name == "finalize_requirement":
            return await self._trigger_matching()
        elif func_name == "clarify":
            return await self._generate_clarification(args["questions"])
```

### 9.5 各阶段输入输出

**阶段一：开场**
```
输入：{environment: {city, device, time_period}, user_message: ""}
输出："你在杭州，用电脑打开的。是跟技术开发、项目外包有关，还是别的？随便说。"
```

**阶段二：意图挖掘**
```
用户："想做小程序卖东西" → 大模型问"卖什么？大概多少种？"
用户："手工皂，二三十种" → update_requirement({project_type:"小程序", industry:"电商", project_nature:"个人创业"})
大模型："明白了，小型手工品自营。在朋友圈卖还是需要完整店铺？"
```

**阶段三：需求展开**
```
后端注入missing_fields和同类参考价，大模型围绕缺失字段自然对话。
大模型不知市价但后端注入，大模型自然融入对话。
```

**阶段四：匹配推荐**
```
后端做匹配，传结果给大模型生成自然语言解释。
输入：{matches: [{name, score, reasons, risk},...], user_requirement_summary}
输出：推荐理由+风险提醒+备选方案，自然融入对话。
```

**阶段五：用户反馈**
```
用户"都不太对" → 大模型先定位问题维度 → 用户选"方向不对"
→ 大模型推断行业假设错了 → update_requirement({industry: "社交"}) → 重新匹配
```

### 9.6 上下文窗口管理

```python
class ContextManager:
    def build_context(self, full_history, max_tokens=8000):
        segments = []
        segments.append(full_history[-6:])       # 最近3轮完整对话
        early_summary = self._summarize(full_history[:-6])
        segments.insert(0, early_summary)         # 早期对话摘要
        segments.append(self.requirement_schema_json)  # 需求Schema(极省token)
        segments.append(self.env_signals_json)    # 环境信号(极省token)
        return segments
```

Token分配(总预算8000)：
- 最近3轮对话: ~2000 | 早期摘要: ~500 | System Prompt: ~1500
- Schema状态: ~300 | 环境信号: ~100 | 缺失字段+引导: ~200 | 留给输出: ~3400

### 9.7 关键Prompt片段

**多轮追问**：围绕missing字段展开对话但不机械按列表提问。"大概""差不多""可能"已足够填字段，不追精确。连续两次回避跳过，用默认假设填充。

**建议生成**：格式为"我建议[明确选择] → 理由1/2 → 如果不同意[备选+代价]"。不写"各有利弊请自行选择"。

**信任建立**：推荐时必须自然带出推荐理由、潜在风险、不满意怎么办。永远不掩盖乙方不完美——诚实比完美更建立信任。

### 9.8 模型选型

| 场景 | 推荐 | 原因 |
|------|------|------|
| 主对话引擎 | Claude Sonnet / GPT-4o | 强推理+多轮一致性+长上下文 |
| 早期对话摘要 | Claude Haiku / GPT-4o-mini | 简单摘要，便宜快速 |
| Function Calling | Claude(首选) / GPT-4 | Claude的tool use结构化抽取更稳定 |
| 中文对话 | Claude Sonnet / 国内模型 | 中文对话质量关键 |

### 9.9 与传统后端协作流程

```
用户消息 → API网关(本地信号采集)
              │
         状态机组装上下文(传统代码)
              │
         大模型调用(理解+生成+Function Call)
              │
      ┌───────┴────────┐
      ▼                ▼
  纯文本回复       Function Call
  (直接返回)       (后端处理)
                      │
                ┌─────┴──────┐
                ▼            ▼
        update_requirement   finalize
        (更新Schema)         (触发匹配→向量检索+规则)
                │                  │
                ▼                  ▼
        检查缺失字段          匹配结果→大模型→推荐解释
        组装新上下文
```

**核心**：大模型只管"理解"和"生成"。所有结构化操作——状态管理、Schema校验、匹配检索、进度判断——都是传统代码，不消耗token。

## 十、最小可行实施方案

只做6个字段、5个维度，最快的上线路径。

### 10.1 甲方需求Schema（6字段版）

```
{
  project_type:    "小程序/APP/网站/后台系统/API服务/其他",
  complexity:      "简单/中等/复杂",
  budget_range:    {min, max},
  timeline_weeks:  number,
  industry:        "电商/教育/金融/社交/企业服务/其他",
  project_nature:  "个人创业/公司项目/side project"
}
```

为什么只留6个：直接切割供给池(project_type)、筛难度级别(complexity)、硬过滤(budget_range)、档期(timeline_weeks)、降低沟通成本(industry)、决定乙方规模(project_nature)。

平台、技术栈、设计、后端、功能清单——靠默认推断覆盖，不追问。

### 10.2 乙方画像（精简版）

```
{
  id, name, type: "个人开发者/工作室/外包公司",
  skills: [{name, projects}],
  service_types: ["小程序","APP","网站",...],
  industry_exp: [{industry, count}],
  complexity_range: {min:1, max:3},
  price_range: {min, max},
  availability: "空闲/较忙/已满", can_start_weeks: number,
  avg_rating, completion_rate, project_count
}
```

### 10.3 五大匹配维度

| 维度 | 权重 | 得分规则 |
|------|------|---------|
| 技能匹配 | 25% | project_type命中→0.6分，复杂度在承受范围→0.4分。不命中→硬过滤 |
| 预算匹配 | 25% | 区间重叠度=重叠/(max(甲方区间,乙方区间))，重叠≤0→硬过滤 |
| 档期匹配 | 15% | 能在时间内启动→1.0，晚1-2周→0.6，晚2周+→0.3，已满→0 |
| 行业经验 | 20% | 同行业≥5项目→1.0，<5→0.7，相邻行业→0.4，无→0.2 |
| 信任/可靠性 | 15% | rating×0.5 + completion_rate×0.3 + log(project_count)/log(50)×0.2 |

### 10.4 权重推断（大模型自动调权）

```json
// 大模型在finalize_requirement时同时返回权重
{
  "summary": "...",
  "weights": {
    "skill_match": 0.20, "budget_match": 0.35,   // ← 用户反复问价格
    "timeline_match": 0.10, "industry_match": 0.15,
    "trust_match": 0.20                            // ← 用户老问"靠谱吗"
  }
}
```

推断规则写入System Prompt：反复提价格→budget_match>0.25，反复提时间→timeline_match>0.20，反复问"靠谱吗"→trust_match>0.20，提行业名词→industry_match>0.20。无特殊信号用默认权重。根据实际表达，每轮可更新一次。

### 10.5 快速启动步骤

1. 建两张表：需求表(6字段) + 乙方表(精简版，手填5-10条测试数据)
2. 写3个Function定义：update_requirement / finalize_requirement(含权重) / clarify
3. 写5维得分函数(每个3-4行) + 加权求和+排序 + 硬过滤
4. 串起来：用户消息→大模型→Function Call→更新Schema→检查完整性→匹配→大模型生成推荐解释
5. 内部测试：模拟5-10段对话，调权重不碰Schema

**跑通标准**：一段完整对话后，Top3里至少有一个"确实合理"。达不到先调匹配得分逻辑，不加字段不加复杂度。
